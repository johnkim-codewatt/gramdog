
## GrammarDog

5차 작업을 진행하게된 이유는 모바일 버전이 필요했기 때문이다. 현재 만들어진 Client, Backend 서비스를 그대로 연동해 붙이는것은 어려운 일이 아니다. 다시 지난 2차 작업때 실패한 모델 학습과 한단계 나아가서 모바일에 llm 모델을 이식하는 것까지 작업해보기로 했다. (우선은 안드로이드기준으로 진행하기로 한다.) 궁극적으로는 현재 서비스에서 호출하는 AI API 를 부분적으로 줄여나가는 것이 목표이다.(기업들의 목표와 동일) 모바일에서 그냥 돌아가게 하는것은 쉬울수 있으나, 잘 돌아가게 하는것은 어렵다. 우선 해당 솔루션에 맞는 모델선택부터, 경량화, 학습 등 고려해야 될점이 많다. 디바이스 상에서 쓸수 있는 부분과 서버에서 써야되는 부분을 잘 배분해야한다. 또한 모바일의 특수성 (메모리, cpu등)을 고려하여 상황별 적절한 라우팅은 필수이다. 

현재는 단계별로 모든 부분에서 api를 호출하고 있다. 당연히 비용이 많이 들어가고, 네트워크 상황에 따라서 답변 생성 딜레이가 길어진다. 기존의 로컬모델 실패를 경험삼아, 더 어려운 더 작은 모델을 활용하여 처리가능한 단계를 디바이스안 모델 안에서 구현한다. 이렇게 했을때는 아래와 같은 장점이 있다.

- **비용 절감 (Cost Efficiency)**:모든 질문에 비싼 API를 쓰는 대신, 모델이 스스로 판단하여 꼭 필요할 때만 외부 자원이나 검색을 수행한다.
- **보안 및 온디바이스 (Privacy & Edge)**: 기업 내부의 민감한 데이터(혹은 개인정보)를 보호하고, 기기에서 독립적으로 구동 가능하게하여 보안성을 높인다.
- **지연 시간 단축 (Low Latency)**: 경량 모델은 응답 속도가 빠르다. 특히 On-demand 검색을 통해 불필요한 데이터 조회를 생략하면 전체 서비스 속도가 향상된다.


## "어느 단계에 적용할것인가?"

<img src="%EC%8A%A4%ED%81%AC%EB%A6%B0%EC%83%B7%202026-04-14%20%EC%98%A4%EC%A0%84%2011.31.10.png" width="304" alt="스크린샷" />

서비스 진행중에 경량 모델이 투입될 자리는 호출2번 단계이다. 문제 생성 단계는 사실 이후에 '데이터증류' 를 통해 '문제은행' 식의 방식으로 교체가 가능하다. 또한 경량 모델의 한계로 인해 적용이 어렵다고 판단되었다. 경량 모델의 한계점은 명확하다. 한마디로 뇌용량이 자체가 작기 때문에 더 똑똑하게 만들수 없다. 이는 레벨/상황/문법에 맞게 다양하고, 정확한 문장을 만들어내기 힘들다고 판단되었다.(된다고 해도 모델 선택과 학습에 대한 비용이 많이 소비됨)


**모델별 차이**

| **구분**    | **LLM (Closed Source)** | **SLM (Open Source)** | **On-device (Encoder/Lightweight)** |
| --------- | ----------------------- | --------------------- | ----------------------------------- |
| **대표 모델** | GPT-4o, Claude 3.5      | Llama 3 (8B), Phi-3   | **KoELECTRA, DeBERTa-v3**           |
| **파라미터**  | 1T+ (추정)                | 3B ~ 8B               | **100M ~ 300M (최적화 타겟)**            |
| **핵심 역할** | 고난도 문제 생성, 데이터 증류       | 파인튜닝용 베이스 모델          | **사용자 발화 분류 및 채점(Step 2)**          |
| **생성 능력** | 문법/상황별 완벽한 생성           | 제한적, 반복적 오류 발생        | **사실상 불가 (분류/회귀 특화)**               |
| **장점**    | 압도적인 데이터 다양성            | 특정 도메인 최적화 가능         | **Zero-latency, Privacy, No Cost**  |
| **한계점**   | API 비용, 보안 리스크          | 온디바이스 구동 시 발열/병목      | **추론 외 복잡한 문맥 파악 한계**               |
|           |                         |                       |                                     |

## 어떤 모델을 이식할것인가?

모델 선택은 사실 환경만 된다면 단순히 큰 모델을 고르는것만으로도 어느정도의 성능이 보장된다. 하지만 모바일이라는 특수한 환경에 맞게 퍼포먼스가 보장되는 가성비 = '경량모델' 을 선택하는것이 중요했다. 몇가지 후보군을 서칭했다.

### 경량 모델 후보
- **gemma4 모델**
		**Gemma 4**는 2026년 4월 2일 구글이 갓 발표한 최신 오픈 모델 시리즈이다. 이전 모델(Gemma 2/3)에 비해 모바일 환경에서의 효율성과 추론 능력이 비약적으로 향상되었다. 모바일 최적화 라인업: E2B & E4B 모델(처음부터 기기 내 배포를 염두에 두고 설계된 E(Edge) 시리즈를 제공한다.)

- **BERT계열** (Embedding) 선정 기준: "정밀한 의미 기반 검색"(검색/매칭/임베딩용)
		문법 오답 데이터나 참고 문서를 검색할 때 GPT와 같은 [[디코더(Decoder)]] 모델보다 BERT와 같은 [[인코더(Encoder)]] 기반 모델(BERT-tiny나 DistilBERT)이 유리하다. 인코더기반의 모델은 쉽게 말해 문맥을 이해하는데 천재이다. (대신 말을 못알아듣고 표현을 못한다.)

- **기타 대표적인 경량모델들**

| **후보 모델**         | **기술적 특징**                  | **선정/제외 사유**                                           |
| ----------------- | --------------------------- | ------------------------------------------------------ |
| **Gemma 4-E2B**   | 2026 최신 Edge 모델, NPU 가속 최적화 | **선정.** Latency와 Memory Footprint의 최적 밸런스.             |

### 베이스 모델 테스트
[[ollama]]를 사용하여 간편하게 후보군 베이스 모델을 테스트해보았다.
``` python 
[role]
너는 최고의 영어학습기의 의도분류기야.
사전단계에서 <출제된 문제>를 기준으로 <사용자 작성>에 대한 의도를 정확하게 분류해야돼.
의도 기준은 아래와 같아.

[의도기준]
- [freetalk] 일상적인대화 및 영어학습과 관계없는 질문
- [unkwoninfo] 알려지지 않은, 잘 모르는 질문
- [translation] 영어학습과 관련된 문장만들기에 해당
- [engstudy] 영어학습과 관련된 사용자의 커스텀 질문

[출제된 문제]
지금 뭐하고 있어?

[사용자 작성]
"오늘 날씨 어때?"

```

Gemma4 모델은 최신답게, 분류작업 정도는 원활하게 진행가능했다. 몇몇 모델은 한글인식등에 어려움이 있어보였다. 해당 파트의 핵심은 한글, 영어를 다 잘해야한다는것이다. (출제된 문제:한글, 유저 입력: 영어 혹은 한글) 따라서 영어기반(원래 잘함)에 한글을 잘하는 모델을 선택해야했다.

![스크린샷](%EC%8A%A4%ED%81%AC%EB%A6%B0%EC%83%B7%202026-04-12%20%EC%98%A4%ED%9B%84%202.31.07.png)


또한 모델을 선택하는것 이외에 모바일이라는 특수한 상황에 맞게 이식작업을 위해서는 일종의 인터페이스가 필요하다. Android, Ios 진영 모두 각각에 맞는 인터페이스를 제공하고 있으므로 내용을 참고해서 진행 해야한다. 

- **Android**라면 `Google AI Edge SDK`
		- **안드로이드** 개발자라면 가장 먼저 고려해야 할 표준입니다.
		- **역할:** 파이썬에서 만든 **Gemma**, Phi-3 같은 모델을 안드로이드 폰에서 직접 실행한다. 복잡한 텐서 연산등을 직접 짤 필요 없이, 구글이 미리 최적화해둔 SDK를 통해 편리하게 연동가능.
		
- **iOS**라면 `CoreML`이나 `MLX-Swift`
		- **MLX-Swift:** 애플 AI 연구팀이 만든 프레임워크로, 파이썬의 PyTorch와 매우 유사한 문법을 Swift에서 구현할 수 있게 해준다. 최신 Gemma 모델을 아이폰의 통합 메모리와 GPU에서 가장 빠르게 실행가능하다.
		- **CoreML:** 파이썬 모델(PyTorch 등)을 `.mlpackage`로 변환하여 아이폰의 뉴럴 엔진(ANE)을 활용하게 한다. 배터리 소모를 최소화하면서 텍스트 임베딩(BERT)이나 분류 작업을 수행할 때 유리하다.



## "왜 인터페이스를 써야하는가?"

기존의 모델과 환경은 파이썬기반으로 설계되어있다. 굉장히 무겁기 때문에 작은 모바일 환경에서 돌리기가 쉽지 않다. 따라서 전용 인터페이스 엔진을 통해 크기를 압축하고(양자화), 최적화하여 모바일환경에서 최적화된 방식으로 사용이 가능하도록 돕는다. 그렇다면, hugging face에 올라온 모델을 다운로드 하면 그대로 올릴수 있는것인가?

일종의 변환작업이 필요하다. 이것도 "뿅" 하고 변환되는 것은 아니고, 개발자가 '변환 스크립트'를 한 번 돌려줘야 한다.

- [[양자화]](Quantization): 이 작업이 핵심인데 이 과정에서 16비트 숫자를 4비트나 8비트로 줄입니다. 그러면 4GB짜리 모델이 1~1.5GB 정도로 줄어듭니다.
- 결과물: 최종적으로 폰이 이해할 수 있는 형태(예: `.tflite`)의 파일이 나옵니다

이렇게 최종적으로 .tflite 확장자로 변환된 모델은, 내부 코드로 호출하는 로직을 구현하면, 모바일에서도 비교적 쉽게? 활용이 가능하다. (말은 쉽다)

1. 변환된 파일을 프로젝트 `assets` 폴더에 넣고, 
2. MediaPipe(Google AI Edge) SDK 를 통해 파일 경로 지정
3. 코딩 상으로는 `llm.generate("오늘 날씨 어때?")` 처럼 함수 호출

### 이슈사항
기존의 Google AI Edge LLM구현 방식(모바일)이 Deplecated되었다. 새로운 방식인 LiteRT-LM을 활용한다.

## "어디에 어떻게 적용할 것인가?"
- **Self-RAG [의도분류] 파트에 적용**
		분류와 같은 단순 작업은 경량모델도 충분히 잘 하는 일이기 때문에, 충분히 교체가 가능하다. 사용자의 의도를 파악하고, 분류 작업을 진행한다. 복합 피드백 처리와 같은 추론이 필요한 부분은 배제한다. 단순 의도 분류뿐만 아니라 해당 모델 수준에서 처리가능한 질문(영어공부와 관련된)이라면 즉시 처리한다. 각 분류는 4가지로 정리한다. 만약 [[인코더]] Encoder 기반의 모델을 베이스로 할것이라면, 미세튜닝을 진행한다.(인코더 기반 모델은 파인튜닝이 필수이다. 말을 못알아듣고, 말을 못한다.)

		[translation, engstudy, freetalk, unknown]
		

- **오답처리 임베딩 검색에 활용**
		'보안' 측면에서  중요한 '개인정보' 인 '유저의 오답 문장' 을 기준으로 device 내에 저장하고, 이것을 [[임베딩]]하고, 조회하는 단계를 대체한다. 단순 임베딩 처리와 의미 기반 서칭이기 때문에 충분히 가능한것으로 판단된다. 임베딩에는 BERT 계열의 Encoder 기반 모델을 선택하자.

### "왜? 인코더 기반 모델을 쓰는가?"
인코더 기반의 모델들이 잘하는 일은 텍스트를 읽고, 이해하고 문맥을 파악하는 능력이다. 물론 최신의 llm (디코더기반) 은 다 잘하지만, 문제는 용량이다. BERT같은 모델은 오래되긴 했지만, 파생된 다양한 종류의 모델이 나와있고 잘 활용하면 세부적인 단계를 쪼개 좋은 성능을 낼수 있다. 

- **BERT (Encoder-only):** 문장을 처음부터 끝까지 **양방향(Bidirectional)**으로 동시에 훑는다. 단어 간의 관계를 입체적으로 파악해서 '의미 덩어리(벡터)'를 만드는 데 특화되어 있으며, "의미가 얼마나 비슷한가?"를 찾는 **검색**에 최적이다.
- **Gemma (Decoder-only):** 문장을 왼쪽에서 오른쪽으로 한 단어씩 예측하며 나아가는 **단방향(Causal)** 구조입니다. 다음 단어를 맞히는 **생성**과 **추론**에는 뛰어나지만, 문장 전체의 의미를 하나의 점(벡터)으로 압축하는 효율은 인코더 모델보다 떨어진다.(그러나 최신의 큰 모델은 다 잘함.)

- **용량 차이:** **BERT-tiny**는 약 **15~30MB** 내외이다. 반면 **Gemma-4-E2B**는 아무리 깎아도 **1GB**가 넘는다.
- **실행 비용:** 유저가 틀린 오답 100개 중에서 유사한 것을 찾을 때, 1GB가 넘는 Gemma를 깨워 연산하는 것보다 20MB짜리 BERT를 상주시켜서 순식간에 검색하는 것이 배터리 소모와 발열 제어 측면에서 압도적으로 유리하다.

## 데이터생성
가장 먼저 할일은 무작정 모델을 이식하는것이 아니라, 학습에 사용할 데이터를 만드는 일이었다. 이 작업에서 가장 적합한 표현은 "Garbage in Garbage out"이다. 데이터가 쓰레기면, 결과도 쓰레기가 나올확률이 높다. 따라서 데이터의 퀄리티가 굉장히 중요하다. 

따라서 실제로 문제 생성시 사용하는 프롬프트를 활용해서, 실제 문제를 만들고 그걸 기반으로 분류 학습에 필요한 데이터를 만들기로 했다.


- **형식:** `text`, `label` 두 개의 컬럼
- **데이터 예시:** `"<q>지금 뭐해?</q> <u>What are...</u>"`, `translation`
- **목표:** 약 1000건의 데이터

데이터 생성시 상당히 어려움을 겪었는데, 처음에는 데이터생성에 대한 AI 비용을 최소화 하기위해 (왜그랬지?) 스크립트를 작성했다. 일종의 랜덤 계수를 이용해 문법, 문장 패턴에 맞게 루프를 돌게 하는것이다. 결과는? 당연히 엉망이었다. 

```
초급, [can/could] + 동사원형, ...
고속버스 터미널 안내소에서 서연은(는) 직원에게 소화기 사용 설명서를 신분증을 보여 줄 수 있는지 공손히 물었다.
```

출제된 문제는 초급이라는 타이틀에 전혀 맞지 않고, 문맥도 엉망이다. 스크립트로 만드는 방식은 몇번을 재시도 했지만, 과감하게 패스하고 넘어갔다. 

다음 방식은 로컬 AI를 활용하는 것이다. 이방법은 Ollama등으로 로컬 환경에 베이스 모델을 이용하여 데이터를 뽑아내는 방법이다. 나름 좋은 방법이었지만, PC 환경이 좋지 못하여 1000건이라는 데이터를 뽑기에 무리가 있었다. 

좋은 방법은 따로 있었다. 그냥 GPT-API를 사용하는것이다. gpt-4o-mini 를 사용해서 데이터를 만들기전에 토큰이 얼마나 소비될지 판단해보았다.

``` markdown
정확한 요금(원가)을 계산해 드릴게요:
- `gpt-4o-mini`는 글자를 만들어내는 데 100만 토큰(단어 조각)당 불과 $0.6 (약 800원)밖에 안 합니다.
- 영작 데이터 1건(예문+정답)을 만들 때 대략 100개의 토큰을 쓴다고 넉넉하게 가정해도, 1,000건이면 고작 10만 토큰입니다.
- 결론: 1,000건을 전부 쏟아내도 한국 돈으로 단돈 '100원~150원' 정도 나옵니다.
```

gpt를 통해 원활하게 데이터를 생성하고, 시간은 약 20분 정도가 걸렸다. 이후에는 데이터 검증을 진행했다. 초급, 중급, 고급별 문법별로 문제출제가 균등하게 잘 작성되었다. (물론 100% 만족은 아니다. MVP 수준에서 이정도면 괜찮은 정도)

이 데이터를 기반으로 학습에 필요한 최종 데이터를 생성한다. 최종데이터는 text, label 두가지 컬럼으로 구성되어있다. 처음에는 최대한 많은 정보를 주고, 이걸기준으로 학습시켜려고 했으나 경량모델 그리고 인코더 기반 분류학습 기준에서는 많은 정보는 노이즈가 될수 있다고 판단했다. 따라서 최적의 학습용 데이터는 "정확한 라벨(문법) + 깔끔한 영어 문장 + 완벽한 한글 번역 쌍" 의 형태로 구성한다.

> *경량 모델(분류작업)은 결국 "특정 텍스트가 들어왔을 때 -> 어떤 문법(Label)으로 분류할지" 를 학습하는 데 목적이 있습니다. 따라서 AI 모델에게 '이건 참고용 예문이고, 이건 상황이 다른 문제야'라고 쪼개서 주는 것은 오히려 노이즈(Noise)만 유발합니다.*

## 데이터 학습
학습은 AI의 도움으로 빠르게 스크립트를 작성한뒤, 최적의 옵션으로 진행했다. 
AutoTokenizer 대신 PreTrainedTokenizerFast 를 사용하여, [[Tokenizer]] 호환성 이슈를 우회(명시적 모델을 사용해 런타임 에러를 방지) 하였다. 

``` python
try:
	# team-lucid/deberta-v3-base-korean 모델은 Fast 토크나이저 구조에 오류,
	# AutoTokenizer 대신 PreTrainedTokenizerFast를 직접 명시하여 우회 로드
	from transformers import PreTrainedTokenizerFast
	tokenizer = PreTrainedTokenizerFast.from_pretrained(MODEL_NAME)
	model = AutoModelForSequenceClassification.from_pretrained(
		MODEL_NAME,
		num_labels=num_labels,
		ignore_mismatched_sizes=True # 레이블 사이즈 충돌을 막는 필수 옵션
	)
	model.to(device)
except Exception as e:
	import traceback
	print("에러 발생:")
	traceback.print_exc()
	sys.exit()
```


1000건은 학습에 있어서 굉장히 적은 양이기 때문에, 데이터 구성에 따라서 일종의 학습 편향이 생길 가능성이 있다. 옵션 조정을 통해 편향을 최소화 했다.
``` python 
train_texts, val_texts, train_labels, val_labels = train_test_split(
texts, labels, test_size=0.2, random_state=42, stratify=labels
)
print(f"📊 학습 데이터: {len(train_texts)}건 | 검증 데이터: {len(val_texts)}건")
```

빠른 학습을 위해 MPS(Metal Performance Shaders) 가속 옵션을 활성화 했다.
``` python 
	# 2. Mac GPU(MPS) 가속 설정
if torch.backends.mps.is_available():
	device = torch.device("mps")
	print("🚀 Apple Silicon GPU (MPS) 가속 활성화")
else:
	device = torch.device("cpu")

```

또한 오버피팅을 방지하고, 검증 단계를 추가하여 최적의 학습단계를 구성하였다. Early Stopping이라는 단계를 구성하여 학습데이터에 대한 내성(오버피팅)시점에 학습을 끊어주는 역할을 한다. 또한 학습 당시 가장 좋았던 학습 성적을 저장해두고, 최종적으로 똑똑했던 시점의 가중치만 저장하는 방식을 적용함으로써 가장 좋은 상태의 학습 모델을 저장한다.
``` python
# ── Early Stopping 체크 ────────────────────────────────
if val_acc > best_val_acc: # (1) 성능 갱신 확인
    best_val_acc = val_acc
    no_improve_cnt = 0
    # 최고 성능 모델을 즉시 저장 (Best Model Save)
    os.makedirs(OUTPUT_DIR, exist_ok=True)
    model.save_pretrained(OUTPUT_DIR) # (2) 가중치 보존
    tokenizer.save_pretrained(OUTPUT_DIR)
    print(f"   💾 최고 성능 모델 저장 (Val Acc: {best_val_acc:.2%})")
else: # (3) 성능 정체 시
    no_improve_cnt += 1
    print(f"   ⚠️  개선 없음 ({no_improve_cnt}/{EARLY_STOP_PATIENCE})")
    if no_improve_cnt >= EARLY_STOP_PATIENCE: # (4) 조기 종료 실행
        print(f"\n🛑 Early Stopping: {EARLY_STOP_PATIENCE}회 연속 개선 없음. 학습 종료.")
        break
```

이렇게 학습된 파일은 아래와 같이 최종 저장된다.
###  파일 리스트
- **생성되는 주요 파일:**
    1. **`config.json`**: 모델의 구조(아키텍처) 정보와 레이블 개수 등이 담긴 설정 파일입니다.
    2. **`model.safetensors`** (또는 `pytorch_model.bin`): 가장 중요한 **학습된 가중치(Weights)** 본체입니다.
    3. **`tokenizer.json` / `tokenizer_config.json`**: 텍스트를 숫자로 바꾸는 규칙(토크나이저) 정보입니다.
    4. **`label_map.txt`**: 숫자로 바뀐 레이블이 원래 어떤 텍스트(예: 0 -> 'grammar_check')였는지 기록한 매핑 파일입니다.


## 모델 변환
학습 완료된 모델파일을 모바일에서 그대로쓸수 없다. LiteRT-LM 사용을 위해서는 .tflite 파일로 변환작업이 필요하다. 변환 작업은 매뉴얼에 따라서 진행하면 비교적 쉽게 진행할수 있다. (파이토치(PyTorch) 모델을 안드로이드에서 돌아가는 **.tflite**로 변환하는 과정)

<img src="%EC%8A%A4%ED%81%AC%EB%A6%B0%EC%83%B7%202026-04-12%20%EC%98%A4%ED%9B%84%2010.32.02.png" width="624" alt="스크린샷" />

``` python
import os
import sys
from pathlib import Path

import torch
import litert_torch
from transformers import AutoConfig, AutoModelForSequenceClassification

# 1. 학습 완료된 PyTorch 모델 불러오기 (pretrain_model.py의 OUTPUT_DIR과 동일)
ROOT = Path(__file__).resolve().parent
MODEL_DIR = str(ROOT / "intent_model_pjt_new")

if not os.path.isdir(MODEL_DIR):
print(f"❌ '{MODEL_DIR}' 폴더를 찾을 수 없습니다. 학습이 정상적으로 완료되었는지 확인하세요.")
sys.exit()


print("📦 로컬 모델을 불러오는 중...")
config = AutoConfig.from_pretrained(MODEL_DIR)
model = AutoModelForSequenceClassification.from_pretrained(MODEL_DIR, config=config)

# 💡 매우 중요: 모델을 '평가(추론) 모드'로 변경
model.eval()

# TFLite/edge 변환은 CPU에서 하는 것이 일반적입니다.
model = model.cpu()

# 2. 가짜 입력(Dummy Input) 생성
# 모바일에서 들어올 데이터와 똑같은 규격(배치 1, 최대 길이 128)의 빈 상자를 만들어 모델을 통과시켜야 합니다.

print("📐 텐서 규격 세팅 중...")
vocab_size = int(config.vocab_size)

sample_input_ids = torch.randint(0, vocab_size, (1, 128), dtype=torch.int64)

sample_attention_mask = torch.ones((1, 128), dtype=torch.int64)

  
# 모델에 들어갈 인자들을 튜플로 묶기
args = (sample_input_ids, sample_attention_mask)


# 3. ai-edge-torch로 TFLite 변환
print("⏳ TFLite 변환을 시작합니다. (맥 미니 성능에 따라 1~3분 정도 소요될 수 있습니다...)")

try:
# PyTorch 모델을 구글의 Edge TPU/CPU 친화적 구조로 변환
edge_model = litert_torch.convert(model, args)

# 4. 파일 추출
output_filename = "intent_classifier.tflite"
edge_model.export(output_filename)
file_size_mb = os.path.getsize(output_filename) / (1024 * 1024)

print(f"\n🎉 변환 대성공! '{output_filename}' 파일이 생성되었습니다.")
print(f"💾 최종 모델 크기: {file_size_mb:.2f} MB")
print("🚀 이제 이 파일을 안드로이드 스튜디오의 'assets' 폴더에 넣으시면 됩니다!")

except Exception as e:
import traceback
print("\n❌ 변환 중 에러 발생:")
traceback.print_exc()
```


만들어진 텐서기반 파일을 변환을 위해 모델을 평가 모드(`eval`)로 전환하고, CPU 기반으로 연산하도록 셋팅하였다. 
```python

model.eval()

# TFLite/edge 변환은 CPU에서 하는 것이 일반적입니다.
model = model.cpu()
```

모바일 NPU/GPU 가속기 활용을 위해 최대길이 128 / 1 로 더미 데이터를 강제로 주입하여, 정적 연산 그래프(Static Graph)를 생성하도록 유도했다.

## 앱개발
앱개발 과정은 초기에 Flutter 기반의 공통 UI구축, 내부에 Native 연동을 위한 channel 을 구축하여 통신하기로 결정했으나, 우선은 안드로이드기반의 MVP를 진행하기로 결정했다. (모델학습 변환 처리등의 과정이 생각보다 복잡하여 안드로이드 이후에 작업을 진행하기로 했다.) 변환된 모델을 사용하는데 있어서 두가지 방식을 선택할수 있다. 

- **`CompiledModel`** API: **고성능** 추론을 위한 최신 표준으로, CPU/GPU/NPU 전반에서 하드웨어 가속을 간소화합니다. [CompiledModel API를 선택해야 하는 이유](https://ai.google.dev/edge/litert/inference?hl=ko)에 대해 자세히 알아보세요.
- **`Interpreter`** API: 하위 호환성을 위해 유지되는 기본 추론 API입니다.


앱개발은 UI에 특별한 기능이 들어가지 않고, 기본 틀 위에 기존의 웹서비스의 프론트엔드 기반의 UI와 유사하게 구현하였다.

모바일 개발단의 복병은 따로 있었다. 바로 '[[토크나이저]]' 의 구현이다. 보통은 파이썬 기반 환경에서는 transformar 라이브러리등을 통해 model, tokenizer등에 대한 처리가 잘 되어 있고, 편리하게 사용할수 있다.

``` python
inputs = tokenizer("개발자는무한학습한다.")
```


반면에 모바일단에서는 모델에 대한 토크나이저 구현을 수동으로 처리해야한다.(물론 구현된 라이브러리등이 있지만, 최대한 구현해보기로 했다.) 이 토크나이저는 굉장히 중요한 요소인데, 인간이 인지 하는 '문자' 를 기계가 인식하는 '숫자' 형태로 어떻게 변환처리 할것인지와, 반대로 '숫자' -> '문자' 로 처리하기 위한 필수 도구이다. '토크나이저' 를 잘못 구현하면(모델 연관) 전혀 다르게 문자를 인식하는 결과가 출련된다. 

> 토크나이저의 3단계 작업 과정
> 토크나이저는 문장이 들어오면 다음 순서로 일을 처리합니다.

1. 텍스트 쪼개기 (Tokenization)** 문장을 모델이 소화하기 쉬운 최소 단위(토큰, Token)로 토막 냅니다. 언어나 모델에 따라 단어, 형태소, 심지어 알파벳 단위로도 쪼갭니다.
	
	- _입력 문장:_ "개발자 개고생"	
	- _쪼갠 결과:_ `["개발", "##자", "무한", "학습", "##한다"]`
	
2. 고유 번호표 부여 (Encoding / ID Mapping)** 모델과 짝을 이루는 거대한 '단어 사전(Vocabulary)'을 뒤져서, 쪼개진 각 조각들에 고유한 정수 번호(Token ID)를 매깁니다.
	
	- "개발" -> `4820`
	- "##자" -> `105`
	- "개" -> `77`
	- "##고생" -> `921`
	- _최종 변환 결과:_ `[4820, 105, 77, 921]`
		
	**3. 특수 기호 추가** 단순히 번호만 주지 않고, 모델이 문장의 시작과 끝을 알 수 있도록 특수한 번호(예: 문장 시작 `[CLS]=101`, 문장 끝 `[SEP]=102`)를 앞뒤에 몰래 끼워 넣습니다.
	
	- _진짜 최종 결과:_ `[101, 4820, 105, 77, 921, 102]`
		
	👉 이제 이 숫자 배열이 비로소 `.tflite` 모델의 입력(Input Tensor)으로 들어갈 수 있게 됩니다. 반대의 과정도 마찬가지이다.


1. Python 모델에서 쓰던 수만 줄짜리 `vocab.txt` (단어 사전 파일)을 앱에 같이 넣고,
2. 사용자가 입력한 문자열을 띄어쓰기나 정규식으로 직접 자르고,
3. 사전을 뒤져서 일일이 숫자로 매핑해 주는 코드를 작성하거나, TensorFlow Lite Support 라이브러리를 연동한다.
   
이런 노가다성 작업은 철저하게 '바이브코딩' 식으로 진행하여 불필요한 시간 소모를 줄였다. 다시 말하지만, 코딩 자체는 이미 인간보다 잘한다. 반드시 해당 모델의 단어사전(vocab.txt와 변환된 모델을 assets에 두고 진행한다.)

``` python
package com.example.aitestapp

import android.content.Context
import java.io.BufferedReader
import java.io.InputStreamReader

class WordPieceTokenizer(context: Context, vocabFileName: String = "vocab.txt") {
    private val vocab = mutableMapOf<String, Long>()
    private val clsTokenId: Long
    private val sepTokenId: Long
    private val padTokenId: Long

    init {
        // assets에서 단어장 로드
        val inputStream = context.assets.open(vocabFileName)
        val reader = BufferedReader(InputStreamReader(inputStream))
        var index = 0L
        reader.forEachLine { line ->
            vocab[line] = index++
        }
        reader.close()
        
        clsTokenId = vocab["[CLS]"] ?: 5L
        sepTokenId = vocab["[SEP]"] ?: 4L
        padTokenId = vocab["[PAD]"] ?: 0L
    }

    // 문장을 숫자로 분해하는 메인 함수
    fun tokenize(text: String, maxLength: Int = 128): Pair<LongArray, LongArray> {
        val tokens = mutableListOf<Long>()
        tokens.add(clsTokenId) // 문장 시작 표시 알림

        // 띄어쓰기 및 임시 특수문자 완벽 분리 (아주 중요: <, >, /, 구두점 등 모두 분리)
        val cleanedText = text.replace(Regex("([^가-힣a-zA-Z0-9])")) { " ${it.value} " }
        val words = cleanedText.trim().split("\\s+".toRegex())

        for (word in words) {
            if (word.isEmpty()) continue
            
            var isBad = false
            var start = 0
            val subTokens = mutableListOf<Long>()
            
            while (start < word.length) {
                var end = word.length
                var curToken: Long? = null
                
                while (start < end) {
                    val substr = word.substring(start, end)
                    val prefix = if (start > 0) "##$substr" else substr
                    if (vocab.containsKey(prefix)) {
                        curToken = vocab[prefix]
                        break
                    }
                    end--
                }
                
                if (curToken == null) {
                    isBad = true
                    break
                }
                
                subTokens.add(curToken)
                start = end
            }
            
            // 모르는 단어 분리 실패 처리
            if (isBad) tokens.add(vocab["[UNK]"] ?: 1L)
            else tokens.addAll(subTokens)
            
            if (tokens.size >= maxLength - 1) break
        }

        tokens.add(sepTokenId) // 문장 종료 알림

        // [128] 크기로 규격 맞추기 (남는 공간은 PAD(0)으로 채우기)
        val inputIds = LongArray(maxLength) { padTokenId }
        val attentionMask = LongArray(maxLength) { 0L }
        
        for (i in 0 until minOf(tokens.size, maxLength)) {
            inputIds[i] = tokens[i]
            attentionMask[i] = 1L // 실제 단어가 있는 곳은 1로 체크해 줍니다
        }
        
        return Pair(inputIds, attentionMask)
    }
}

```


BERT/DeBERTa 계열의 핵심인 [[WordPiece 알고리즘]] 을 Kotlin으로 이식하여 처리했다. 단어를 더 이상 쪼개지지 않는 최소 단위(Subwords)로 분해하며, 특히 `##` 접두사를 처리하는 로직을 통해 학습 시 사용된 토크나이저와의 알고리즘 일관성(Consistency)을 확보해야한다. 

정규식으로 전처리하여 한국어(가-힣), 영문, 숫자를 제외한 모든 특수문자와 구두점을 토크나이징 전 단계에서 완벽히 분리함. 이는 영어 학습 앱 특성상 마침표(.), 물음표(?) 등 문장 부호가 문맥 파악에 주는 임팩트를 정확히 모델에 전달하기 위한 전처리 설계이다.

``` python
// 띄어쓰기 및 임시 특수문자 완벽 분리 (아주 중요: <, >, /, 구두점 등 모두 분리)
val cleanedText = text.replace(Regex("([^가-힣a-zA-Z0-9])")) { " ${it.value} " }
```

앞선 파이썬 변환 단계에서 정의한 정적 입력 규격(1, 128)에 맞추기 위해 패딩(PAD) 처리를 수행함. 특히 실제 데이터가 있는 영역을 1로 표시하는 `attentionMask`를 생성하여, 모델이 패딩된 무의미한 영역을 연산에서 제외하도록 설계한다. 

> AI 모델의 성능은 학습 시 사용한 토크나이저와 배포 환경의 토크나이저가 얼마나 일치하느냐에 달려 있다.(이쪽 따로 저쪽 따로 하면 안됨) 본 구현체는 HuggingFace의 `PreTrainedTokenizerFast`와 **동일한 특수 토큰([CLS], [SEP], [UNK]) 체계 및 WordPiece 로직**을 따르도록 설계되어, 파이썬 환경의 성능을 안드로이드 기기에서도 손실 없이 재현하도록 보장한다.



## 결과
최종 작업된 결과는 대성공? 이었다. 분류/정답오답 판단이라는 세분화된 영역을 분류하는 작업을 타겟으로 했고, 이런 작업은 경량모델도 훌륭하게 해낼수 있는 부분이다. 좋은 모델을 선택했고, 좋은 데이터를 통해 학습시켰다. 이럴경우 당연히 결과가 잘나올 확률이 높다.

### **파인튜닝 전략 및 데이터 엔지니어링**

서비스 최적화를 위해 파인튜닝(Fine-tuning) 영역까지 작업 범위를 확장한다. 주요 목적은 추론 비용 절감과 기술적 내재화(학습)에 있으며, 상용 모델에만 의존하기보다 특정 도메인에 특화된 모델을 구축함으로써 운영 효율을 극대화하고자 한다.

모델 튜닝의 성패를 좌우하는 것은 결국 데이터의 품질이다. 'Garbage In, Garbage Out' 이라는 격언은 가장 본질적인 진리이다. 양질의 데이터가 전제되지 않은 학습은 무의미하기 때문이다. 문제는 실전 데이터가 없다는것이다. 고성능의 LLM(GPT-4o or mini)을 활용한 **'[[데이터 증류]](Data Distillation)'** 작업을 진행했다.

이를 통해 최소 500~$1,000$건의 학습 데이터를 생성하되, 정답 데이터와 오답 데이터의 비율을 특정 편향에 빠지지 않게 $1:1$ 로 구성한다. 특히 영작 교정 서비스의 특성상 단순 정답보다는 '어떤 오류가 발생하는지'를 보여주는 **부정 데이터(Negative Data)**의 질과 양이 모델의 판별 능력을 결정하는 핵심 변수임을 인지하고 이를 정밀하게 설계한다. 누가 봐도 틀린 문장보다는, 실제 사용자가 자주 헷갈리는 미묘한 시제 오류나 관사 누락 같은 '어려운 오답'을 배치하는 것이 모델의 변별력을 높인다. 또한 특정 오답 패턴만 너무 반복적으로 부정 데이터에 넣으면, 모델은 해당 단어만 보이면 무조건 틀렸다고 간주하는 '앵무새 현상' (과적합) 에 빠지는것을 주의해야한다.

### **데이터 비율**
*정답 데이터만 너무 많으면 모델이 모든 입력을 '정상'으로 간주하여 교정 능력이 떨어지고, 반대로 부정 데이터가 너무 많으면 멀쩡한 문장까지 수정하려 드는 '과교정(Over-correction)' 현상이 발생한다.*

| **태스크 유형**               | **정답 : 부정 비율**    | **이유**                                            |
| ------------------------ | ----------------- | ------------------------------------------------- |
| **단순 명령어 이행(SFT)**       | $1:0$ (100% 정답)   | 모델에게 올바른 수행 방식(Gold standard)만 학습시킨다.             |
| **문법 교정 및 감사(Critique)** | **$1:1$ ~ $1:3$** | 오답의 패턴이 다양할수록 '무엇이 틀렸는지'를 배우는 데 유리하다.             |
| **선호도 학습(DPO/RLHF)**     | $1:1$ (Pair)      | 하나의 질문에 대해 '좋은 답변'과 '나쁜 답변'을 쌍으로 묶어 학습한다.         |
| **검색 및 매칭(RAG)**         | $1:5$ 이상          | 수많은 오답(Hard Negatives) 사이에서 단 하나의 정답을 찾는 능력을 키운다. |
|                          |                   |                                                   |


``` python
import openai
import json
import random

client = openai.OpenAI(api_key="asdfasdfasdf123123123")

# 앱 기획에 맞춘 설정값
categories = ["일상 생활", "비즈니스/업무", "해외 여행", "학술/토론", "취업/인터뷰"]
levels = ["초급 (Beginner)", "중급 (Intermediate)", "고급 (Advanced)"]
grammars = ["현재완료", "관계대명사", "가정법", "동사 시제 일치", "관사 사용", "전치사"]

def generate_question_data(count=10):
    dataset = []
    for i in range(count):
        category = random.choice(categories)
        level = random.choice(levels)
        grammar = random.choice(grammars)
        
        system_prompt = "너는 영어 교육용 문제를 출제하는 전문 출제위원이야."
        user_prompt = f"""
        다음 조건에 맞는 영어 학습용 번역 문제를 생성해서 JSON으로 출력해줘.
        
        [조건]
        1. 카테고리: {category}
        2. 난이도: {level} (난이도에 맞는 어휘와 문장 길이를 엄격히 준수할 것)
        3. 학습 목표 문법: {grammar}
        
        [출력 형식]
        {{
          "instruction": "유저의 레벨과 요청된 주제에 맞춰 적절한 상황을 부여하고, 번역할 한글 문장을 제시하세요.",
          "input": {{
            "category": "{category}",
            "level": "{level}",
            "target_grammar": "{grammar}"
          }},
          "output": {{
            "situation": "영작이 필요한 구체적인 상황 설명 (한국어)",
            "korean_sentence": "유저가 영작해야 할 한글 문장",
            "english_answer": "가장 표준적인 모범 영작 답안",
            "key_vocabulary": ["문장에 사용된 주요 단어", "뜻"],
            "grading_points": "{grammar}를 어떻게 활용해야 하는지에 대한 출제 포인트"
          }}
        }}
        """

        response = client.chat.completions.create(
            model="gpt-4o", # 비싼모델 사용
            messages=[
                {"role": "system", "content": system_prompt},
                {"role": "user", "content": user_prompt}
            ],
            response_format={ "type": "json_object" }
        )

        data = json.loads(response.choices[0].message.content)
        dataset.append(data)
        print(f"[{i+1}/{count}] 문제 생성 완료: {level} - {grammar}")

    return dataset

# 실행 및 저장
question_data = generate_question_data(50) 
with open("question_gen_data.jsonl", "w", encoding="utf-8") as f:
    for entry in question_data:
        f.write(json.dumps(entry, ensure_ascii=False) + "\n")

print("🎉 문제 생성 데이터가 question_gen_data.jsonl로 저장되었습니다!")
```

>*구글 코랩(Google Colab)은 클라우드환경에서 파이썬(Python) 코드를 작성하고 실행할 수 있도록 하는 Jupyter Notebook 환경입니다. 구글 클라우드 인프라를 활용하여 하드웨어 자원을 할당받기 때문에, 개인 PC 사양에 구애받지 않고 고성능 컴퓨팅 작업이 가능합니다.*

학습 효율을 극대화하기 위해 **[[Unsloth]]** 와 **[[LoRA]](Low-Rank Adaptation)** 기술을 결합하여 사용한다. Unsloth는 모델 아키텍처에 최적화된 수동 커널을 사용하여 메모리 사용량을 70% 이상 절감하고 학습 속도를 비약적으로 향상시키는 최적화 라이브러리이다. 이를 통해 제한된 리소스 환경에서도 모델 [[양자화]](Quantization) 및 다운사이징을 실현하여 원활한 학습을 도모한다.

>Unsloth 는 거대 언어 모델(LLM)을 파인튜닝할 때 속도는 2~5배 더 빠르게, 메모리 사용량은 70% 이상 줄여주는 오픈소스 최적화 라이브러리입니다. 개인 개발자가 코랩(Colab) 같은 제한된 환경에서 상용급 모델을 구울 수 있게 해주는 '치트키'와 같습니다. 한마디로 "PyTorch의 범용 Autograd(자동 미분) 시스템을 특정 모델 아키텍처(Llama, Mistral 등)에 최적화된 수동 커널(Triton 기반)로 대체한 라이브러리"입니다.

비유하자면 표준적인 학습 방식이 "두꺼운 전공 서적 전체를 통째로 복사하는 것"이라면, Unsloth는 "중요한 핵심 요점만 형광펜으로 칠해서 초고속 스캐너로 읽어들이는 것" 이다. (주의: 당연히 로스가 발생할것이다) Unsloth 와 같이 많이 쓰는 기술이 LoRA(Low-Rank Adaptation) 이다. LoRA 학습 방식은 모델 전체를 수정하는 대신, 특정 지침이 담긴 얇은 '지식 레이어(Adapter)'를 베이스 모델 위에 덧씌우는 형태를 취하는 방식이다. 우리가 준비한 데이터를 학습할 때 모델 전체를 수정하는 게 아니라, 아주 얇은 '지식 레이어' 만 새로 만드는 것이다. 이렇게 하면 일종의 필터, 마스크, 어댑터가 만들어지고 베이스 모델은 그대로 두고 어댑터만 갈아끼우면 되는 최적의 방식이다.

1. **Base Model:** Llama 3.3 8B 같은 거대한 기본 뇌 (지식은 많지만 튜터 말투는 모름)
2. **LoRA Adapter:** 우리가 뽑은 50개의 데이터 를 학습하여 만들어진 '필터' 
3. **Questioner LoRA:** "상황 설명"과 "한글 문장"을 만드는 법을 학습
4. **Tutor LoRA:** "오답 이력"을 참고해 "격려 섞인 피드백"을 주는 법 학습

해당 기술을 활용하기 위해 한가지가 더 필요하다. 바로 Hugging Face Endpoint 서비스이다. Hugging Face는 전세계의 수 많은 AI 모델을 공유하는 오픈소스플랫폼이다. Hugging Face는 전 세계의 다양한 AI 모델이 공유되는 오픈소스 플랫폼으로, 각 도메인에 특화된 수천 가지의 모델을 제공한다. 서비스의 목적과 규모에 부합하는 최적의 베이스 모델을 식별하고 선택하는 과정은 전체 시스템 설계의 성패를 좌우하는 핵심적인 시작점이다.


### Endpoint 셋팅
비교적 경량 모델인 llama3.x 8B를 선택하고, 저장소를 만든다. 사전에 만들어둔 학습 데이터인 'adapter' 파일을 업로드 해준다. 

베이스 모델로 **Llama-3.1-8B**를 선정한 것은 성능과 자원 효율성 사이의 최적점(Sweet Spot)을 찾기 위한 전략적 판단이다. 70B 이상의 대형 모델은 정교한 추론 능력을 보장하지만, 실시간 서비스 서빙 시 발생하는 막대한 인프라 비용과 응답 지연(Latency) 문제를 무시할 수 없다. 반면 8B 체급은 단일 GPU 환경(VRAM 24GB 내외)에서도 안정적인 운용이 가능하며, Unsloth와 LoRA를 통한 경량화 학습 시 가장 뛰어난 효율을 보여주는 '골든 스탠다드' 모델이기 때문이다.

**체급별 성능 및 자원 효율성 비교**

| **구분**     | **소형 모델 (1B~3B)**                               | **중형 모델 (8B~12B)**                                       | **대형 모델 (70B 이상)**                            |
| ---------- | ----------------------------------------------- | -------------------------------------------------------- | --------------------------------------------- |
| **추론 능력**  | 기초적인 단어 추출은 가능하나, 복잡한 영문법 논리 및 시제 분석 능력이 현저히 낮음 | **언어 추론과 문맥 이해의 균형이 우수하며, 128K의 확장된 컨텍스트 윈도우 지원**        | 최상급의 추론 능력을 보유하나, 서비스 응답 속도(Latency)가 매우 느림   |
| **메모리 점유** | 1~2GB 수준으로 매우 가벼움                               | **4-bit 양자화 시 약 5.5GB 점유. 24GB VRAM 환경에서 여유 자원 충분**      | FP16 기준 140GB 이상, 양자화 시에도 최소 40GB 이상의 VRAM 요구 |
| **운용 비용**  | 최저 수준                                           | **시간당 약 $0.5 ~ $1 (Hugging Face Endpoint 기준)로 상용 서빙 가능** | 다중 GPU 노드(A100/H100 등)가 필수적이어서 수십 배 이상의 비용 발생 |

- **Qwen-2.5-7B와의 비교:** Qwen은 논리 추론 및 한국어 이해도가 뛰어나지만, Llama-3.1은 전 세계적으로 가장 방대한 튜닝 데이터와 **Unsloth** 같은 최적화 라이브러리의 전폭적인 지원을 받는다. 이는 개발 초기 단계에서 시행착오를 줄이고 빠르게 안정적인 결과물을 도출하는 데 결정적인 요소이다.

- **Mistral-Nemo-12B와의 비교:** 12B 모델은 '체급이 깡패'라는 말처럼 지능 면에서 8B를 앞서기도 하지만, 4-bit 양자화 적용 시 T4 GPU(16GB VRAM) 환경에서 메모리 한계치에 도달하여 안정성이 저하된다. 반면 Llama 8B는 가용 자원 내에서 멀티 로라(Multi-LoRA) 어댑터를 동시에 적재하고도 충분한 처리량(Throughput)을 확보할 수 있다.

## 어디서 어떻게 쓸것인가?
셋팅된 Endpoint는 24GB GPU 를 마음껏 호출할수 있다. ($10 만큼) 본 프로젝트에서는 '문제출제자', '피드백시점' 에 활용하기 위해 두가지 호출 포인트를 만들어야 했다. 따로 저장소를 구분하여 모델을 구성할수도있고, 1모델에 LoRA 기술을 적용하면, 아래의 셋팅만으로 각각의 파라미터 호출시 어댑터를 갈아끼우는 방식으로 처리할수 있다. 

```
--max-model-len 4096 --gpu-memory-utilization 0.85 --max-num-seqs 32 --enable-lora --max-loras 2 --lora-modules questioner=demouser1/llama-3.1-eng-questioner tutor=demouser1/llama-3.1-eng-tutor
```

	**중요**
	중요한 포인트는 무한과금이 되지 않도록 최소응답시간을 15분으로 맞춰야 한다.

### **결과 및 기술적 반성 (Results & Technical Post-mortem)**

모델 학습과 클라우드 인프라를 연계한 1차 테스트 결과는 기대에 미치지 못했다. 기존 **GPT-4o**를 파인튜닝 모델로 교체하여 성능을 검증했으나, 실무에 적용하기 어려운 수준의 결과물이라 판단하였다. 특히 기존 1:1에서 벗어나, $1,000$건의 데이터($200$건의 정답 및 $800$건의 오답)를 학습시킨 후 발생한 현상은 전형적인 **과적합(Overfitting)**과 **지능의 저하**였다. 

가장 핵심적인 패착은 **모델의 체급과 태스크 복잡도 사이의 상관관계, 즉 적용 파트에 대한 이해 부족**이다. 경량 모델을 활용하면서 단일 단계에 고도의 복잡한 추론을 요구한 것이 실패의 근본 원인이다.

경량 모델의 한계를 극복하려면 성능을 최적화하기 위해서 **로직을 최대한 세분화하여 단계적으로 적용**하는 전략이 필수적인 것이다. 단일 프롬프트에 모든 분석과 교정을 맡기는 대신, 태스크를 최소 단위로 쪼개고 각 단계마다 모델이 집중해야 할 목표를 명확히 부여해야 한다. 이러한 단계별 구조화(Step-by-step Decomposition)는 모델의 추론 부하를 분산시켜, 낮은 체급의 모델로도 상용급 결과물을 도출할 수 있게 만드는 핵심 기법이다.

- **과적합으로 인한 유연성 상실:** 동일하거나 유사한 패턴의 데이터를 반복 학습한 결과, 모델이 특정 문구만을 되풀이하는 이른바 '앵무새 현상'이 발생한다. 이는 프롬프팅으로 제어할 수 있는 범위를 넘어서며, 모델이 가진 본래의 추론 능력을 억제하는 결과로 이어진다.
    
- **학습 프로세스의 중복성:** 본 프로젝트의 핵심인 영문법 및 어휘 지식은 이미 베이스 모델에 충분히 내재되어 있다. 사전에 과할 정도로 학습된 범용 지식을 다시 튜닝하는 과정은 실익이 낮을 뿐만 아니라, 오히려 모델의 판단 기준을 혼란스럽게 만드는 독이 된다.
    
- **모델 체급에 따른 추론 능력의 한계:** 사용자가 작성한 영작문을 정밀하게 분석하고 교정하는 작업은 고도의 논리적 추론을 요구한다. **8B** 체급의 경량 모델은 뇌 용량의 물리적 한계로 인해 이러한 복잡한 분석을 수행하기에 기초 지능이 부족하다. 이는 **LoRA**와 같은 외부 레이어 추가만으로는 극복하기 어려운 영역임을 확인한다.
    
결과적으로 단순히 모델을 '굽는' 행위보다, 해결하고자 하는 문제의 복잡도에 맞는 베이스 모델의 선정(Base Model Selection)이 더 중요하다는 결론에 도달했다. 이에 따라 무리한 파인튜닝을 멈추고, 각기 다른 특성을 가진 오픈소스 베이스 모델들을 대상으로 비교 테스트를 진행하여 최적의 대안을 모색하기로 하였다.

## "그렇다면 학습은 필요없는것인가?"

### **파인튜닝을 지양해야 하는 경우 (Anti-patterns)**
모델의 말투를 바꾸거나 JSON 형식을 강제하는 등의 지시사항은 시스템 프롬프트(System Prompt)만으로도 충분히 구현 가능하다. 이를 학습으로 해결하려 할 경우, 모델이 특정 패턴에만 고착되는 과적합(Overfitting)이 발생하여 전반적인 추론 지능이 저하되는 역효과가 발생할 우려가 있다.
    
오타 탐지나 시제 파악과 같은 근본적인 추론 능력은 학습을 통해 주입하기 어렵다. 이는 모델의 '기초 체력' 문제로, 학습보다는 더 똑똑한 베이스 모델로 교체하는 것이 훨씬 경제적이고 확실한 해결책이다.

### **파인튜닝이 필요한 결정적 순간 (Triggers)**
범용 모델은 범용 모델이다. 특정 분야에 특화된 지식이 필요한 경우 학습이 필요하다. 예를 들어 한국인 학습자가 습관적으로 사용하는 '콩글리시' 패턴(예: night work → worked overtime)을 사전에 걸러내고 싶을때 활용하면 된다. 이러한 방대한 오답 노트 패턴을 프롬프트에 나열하는 것은 비효율적이고, 퍼포먼스가 저하되는 방법이다. 특정 패턴 1,000건의 특화 데이터를 학습시켜 모델의 뇌에 직접 지식을 이식하는 방식을 취한다. 또한 학습없이 퓨샷(Few-shot) 예시를 늘릴수록 입력 토큰이 증가하여 추론 속도가 저하되고 비용이 상승한다. 이런경우에도 파인튜닝을 통해 특정 포맷과 지식을 모델의 가중치에 내재화하면, 긴 예시 없이도 빠르고 정교한 답변을 얻을 수 있어 상용 서비스 운영 시 유리하다.

### **실무적 관점의 '태스크 특화 모델(Task-Specific Model)' 전략**
실제 상용 환경에서는 비용 효율을 위해 거대 모델(GPT-4 등) 대신 **1B~2B급의 소형 모델을 특정 태스크에 특화시켜 활용**하는 일종의 꼼수가 유효하게 작용한다. 예를 들어, 영수증 파싱과 같은 좁은 범위의 업무는 만 건 이상의 데이터를 학습한 소형 모델이 대형 모델보다 빠르고 정확하게 수행할 수 있다.(경량모델은 경량모델의 장점이 있다.)

본 프로젝트에서 **Llama 8B** 학습이 실패했던 이유는 모델의 '기본적 눈치(추론력)'가 부족한 상태에서 응용을 강요했기 때문이라 판단된다. 사실상 언어는 조합이 무한하며 높은 추론 능력이 뒷받침되어야 하는 영역이다. 따라서 기초 지능이 더 뛰어난 모델을 기반으로 특화 데이터를 학습시킨다면, 부족한 지식은 학습으로 채우고 복잡한 분석은 모델의 자체 지능으로 처리할수 있는 영역이라 판단한다.
